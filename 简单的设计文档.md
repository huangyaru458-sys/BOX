# 智能衣物回收箱物联网系统设计说明

## 1. 项目概述
本项目面向社区旧衣物回收与设备远程运维场景，设计一套基于 ESP32 的智能衣物回收箱物联网系统。系统以传感器采集、无线传输、后端处理、Web 展示、移动端管理为主线，结合离线 AP 应急模式，实现回收箱状态监测、自动投放控制、异常告警、远程运维与历史数据查询。

项目主控统一采用 ESP32，不使用树莓派。系统主流程采用 `ESP32 + MQTT + Node-RED + Web + uni-app` 的结构，满足实训要求中的感知层、网络层、应用层完整链路。

## 2. 需求分析

### 2.1 应用场景与目标用户
本系统面向社区旧衣物回收箱、校园回收点、物业服务点等实际投放场景，用于实现回收箱状态监测、远程运维、异常告警和本地应急管理。

目标用户分为两类：
1. 运维管理员：负责查看设备运行状态、接收异常告警、执行远程开锁和风扇控制、进行设备巡检。
2. 实训开发人员：负责嵌入式采集、后端服务、Web 展示、手机端开发与联调测试。

### 2.2 系统需求目标
1. 让回收箱具备自动感知能力，能够识别人员靠近、箱体满载、空气异常和倾倒风险。
2. 让回收箱具备自动联动能力，能够根据传感器状态控制门体、风扇、蜂鸣器和指示灯。
3. 让系统具备远程管理能力，能够通过 Web 端和手机端查看状态、查询历史和下发控制。
4. 让系统具备数据留痕能力，能够将实时数据、历史数据、告警数据和控制记录统一保存。
5. 让系统具备断网应急能力，后端不可用时仍可通过 ESP32 热点完成本地查询与开锁。

### 2.3 功能需求

#### 2.3.1 传感器感知与唤醒需求
1. 人体红外传感器 HC-SR501 作为投放唤醒入口，当检测到人员靠近时，唤醒投放流程，触发投放门开启逻辑。
2. 超声波传感器 HC-SR04 用于检测箱内剩余空间，通过测距结果判断满载率，并把满载判断结果同步给后端和前端。
3. 温湿度传感器 DHT11 用于采集箱体内部温湿度，作为环境监测和防霉判断依据。
4. 空气质量传感器 MQ-135 用于检测箱内异味或气体浓度，异常时触发通风联动和告警。
5. 姿态传感器 MPU6050 用于检测箱体倾斜、倾倒等防盗状态，异常时进入告警锁定流程。
6. 传感器数据采集完成后，ESP32 将数据封装为统一格式上报后端，同时驱动本地 OLED 和状态灯刷新显示。

#### 2.3.2 本地控制需求
1. 本地管理员可通过 4×4 矩阵键盘输入密码进行维护门解锁。
2. 本地控制优先级高于普通远程查看，但受满载锁定和防盗锁定规则约束。
3. 本地控制触发后，ESP32 需同步更新 OLED 显示、LED 状态灯和蜂鸣器状态。
4. 本地控制记录需写入控制日志，并在网络恢复后同步到后端。

#### 2.3.3 远程控制需求
1. Web 端和手机端可通过后端发起远程开门、关门、启停风扇、取消告警等操作。
2. 远程控制指令由后端统一校验后转发到 ESP32，避免前端直接操作设备。
3. 远程控制请求必须带有设备编号和管理员权限信息。
4. 远程控制结果需通过设备状态回传后同步更新前端界面。

#### 2.3.4 数据存储需求
1. 系统必须保存实时状态数据，用于首页展示和设备状态刷新。
2. 系统必须保存历史时序数据，用于趋势分析和图表展示。
3. 系统必须保存告警数据，用于事件追溯和运维处理。
4. 系统必须保存控制记录，用于审计和问题排查。

#### 2.3.5 查询与可视化需求
1. Web 端需要展示设备实时状态、历史曲线、告警列表和控制面板。
2. 手机端需要展示实时卡片、趋势图、告警消息和远程控制按钮。
3. 查询结果必须支持按设备编号和时间范围检索。
4. 历史数据应支持图表方式展示，重点展示温湿度、空气质量和容量变化趋势。

#### 2.3.6 后端服务需求
1. 后端接收 ESP32 通过 MQTT 上报的数据并进行解析、校验和入库。
2. 后端向 Web 端和手机端提供 HTTP API。
3. 后端维护设备在线状态和最后上报时间。
4. 后端将控制请求转发给设备，并记录执行结果。
5. 后端对异常数据进行过滤，对重复告警进行去重处理。

#### 2.3.7 Web 端软件需求
1. Web 端应具备设备总览页面，显示设备编号、在线状态和最新运行数据。
2. Web 端应具备历史数据页面，显示 24 小时趋势曲线。
3. Web 端应具备控制页面，提供远程开门、关门、风扇控制和告警确认。
4. Web 端应具备告警页面，用于查看当前告警和历史告警记录。

#### 2.3.8 手机端软件需求
1. 手机端应具备移动监控首页，优先显示设备实时状态。
2. 手机端应具备历史趋势页，支持按时间查看图表。
3. 手机端应具备远程控制页，支持管理员执行开锁和风扇控制。
4. 手机端应具备离线模式页，在连接 ESP32 热点后查看本地状态和执行本地开锁。

### 2.4 硬件需求
系统至少需要以下硬件：

#### 2.4.1 主控与通信硬件
1. ESP32 开发板：作为系统主控，负责采集、计算、通信和控制。
2. WiFi 模块：ESP32 自带，用于连接后端和提供 AP 热点。

#### 2.4.2 传感器
1. DHT11 温湿度传感器：采集箱体内部温湿度。
2. MQ-135 空气质量传感器：采集箱内空气质量或异味浓度。
3. HC-SR04 超声波传感器：检测箱体余量并计算满载率。
4. HC-SR501 人体红外传感器：检测人员靠近并唤醒投放流程。
5. MPU6050 姿态传感器：检测倾斜和倾倒风险。

#### 2.4.3 执行器
1. OLED 显示屏：本地显示实时数据、告警和网络状态。
2. 门体执行器（可选）：用于投放门或维护门开启关闭。
3. L298N 电机驱动模块：控制直流风扇启停，实现箱体通风。
4. 蜂鸣器：用于声光告警。
5. LED 指示灯：用于显示正常、预警、故障等状态。

### 2.5 硬件接线与引脚分配
#### 2.5.1 供电与共地规则
1. ESP32 由 USB 供电或开发板自带稳压供电。
2. OLED、MPU6050、蜂鸣器使用 3.3V 供电。
3. DHT11、MQ-135、HC-SR04、HC-SR501、L298N 逻辑侧可按模块要求接 5V。
4. 风扇使用独立外部电源，风扇电源负极必须与 ESP32 共地。
5. 所有模块 GND 必须与 ESP32 GND 连通，禁止悬空。

#### 2.5.2 完整接线表
| 模块 | 引脚说明 | ESP32 引脚 | 备注 |
|---|---|---|---|
| DHT11 温湿度传感器 | VCC、GND、DATA | `DATA -> GPIO17` | 采集箱内温湿度 |
| MQ-135 空气质量传感器 | VCC、GND、AO | `AO -> GPIO34` | ADC 采集，AO 接模拟口 |
| HC-SR04 超声波传感器 | VCC、GND、Trig、Echo | `Trig -> GPIO2`，`Echo -> GPIO4` | 用于满载率计算 |
| HC-SR501 人体红外传感器 | VCC、GND、OUT | `OUT -> GPIO16` | 人体靠近唤醒 |
| MPU6050 姿态传感器 | VCC、GND、SDA、SCL | `SDA -> GPIO21`，`SCL -> GPIO22` | 与 OLED 共用 I2C |
| OLED 显示屏 | VCC、GND、SDA、SCL | `SDA -> GPIO21`，`SCL -> GPIO22` | 本地状态显示 |
| 4×4 矩阵键盘 | R1、R2、R3、R4、C1、C2、C3、C4 | `R1=GPIO18`，`R2=GPIO19`，`R3=GPIO23`，`R4=GPIO32`，`C1=GPIO25`，`C2=GPIO26`，`C3=GPIO13`，`C4=GPIO33` | 本地密码输入 |
| 蜂鸣器 | VCC、GND、IO | `IO -> GPIO5` | 声光告警 |
| LED 指示灯 1 | 正极、负极 | `GPIO14` | 建议作为正常状态灯 |
| LED 指示灯 2 | 正极、负极 | `GPIO12` | 建议作为告警/故障灯 |
| L298N 电机驱动模块 | ENA、IN1、IN2、OUT1、OUT2、12V/VIN、GND | `ENA -> GPIO27`，`IN1 -> GPIO15`，`IN2 -> GND` | 控制风扇启停，ENA 可做 PWM |
| 直流风扇 | 两根电机线 | 接 L298N `OUT1/OUT2` | 风扇由 L298N 驱动 |

#### 2.5.3 风扇驱动接线说明
1. L298N `12V/VIN` 接风扇外部电源正极，`GND` 接风扇外部电源负极。
2. L298N `GND` 与 ESP32 `GND` 必须共地。
3. 风扇电机两根线接到 L298N 的 `OUT1/OUT2`。
4. L298N `ENA` 接 ESP32 `GPIO27`，用于使能和 PWM 调速。
5. L298N `IN1` 接 ESP32 `GPIO15`，用于风扇启停。
6. L298N `IN2` 直接接 `GND`，固定为低电平。
7. 如果 L298N 板载跳帽和 5V 逻辑供电冲突，以模块说明书为准，原则是控制侧和电机侧电源分开、共地连接。

常见控制逻辑：
1. `GPIO15=1, GPIO27=1`：风扇开启。
2. `GPIO15=0, GPIO27=0`：风扇停止。
3. 若需要 PWM 调速，可在 `GPIO27` 上输出占空比信号。

### 2.5 软件功能与交互需求
#### 2.5.1 嵌入式与传感器交互
1. ESP32 周期读取 DHT11、MQ-135、HC-SR04、HC-SR501、MPU6050 的状态。
2. HC-SR501 触发后，系统进入投放唤醒状态，打开投放门并提示可投放。
3. HC-SR04、DHT11、MQ-135 和 MPU6050 的数据统一封装后发送给后端。
4. 当满载、异味超标或倾倒异常时，ESP32 本地先执行声光提示，再同步上报后端。

#### 2.5.2 嵌入式与 Web/App 交互
1. Web 端和 App 端通过后端接口轮询实时状态。
2. 后端将最新状态返回给前端后，前端刷新卡片、图表和告警信息。
3. 当用户点击控制按钮时，请求先到后端，再由后端转发到 ESP32。
4. 控制结果通过设备状态回传更新前端，前端无需直接连接硬件。

#### 2.5.3 本地控制与远程控制协作
1. 本地密码输入成功后，可执行维护门开锁。
2. 远程控制用于管理员在 Web 或 App 上远程操作设备。
3. 本地模式和远程模式共用一套状态机，避免控制冲突。
4. 满载锁定、倾倒告警等状态具有更高优先级，可覆盖普通开门请求。

### 2.6 非功能需求
1. 实时性：采集周期约 2s，常用控制指令响应应控制在 1s 内。
2. 稳定性：系统应支持长时间连续运行，避免任务阻塞和通信死锁。
3. 容错性：单个传感器异常不应导致整个系统失效。
4. 可维护性：模块划分清晰，便于嵌入式、后端、前端分别开发和调试。
5. 兼容性：支持 PC 浏览器和移动端访问，支持局域网和外网两种使用方式。
6. 可扩展性：后续可增加更多传感器、更多控制策略和更多告警规则。

## 3. 系统总体设计

### 3.1 分层架构
系统采用三层物联网架构，并在应用层细分 Web 端和移动端：

1. 感知控制层：ESP32、传感器、执行器、本地显示与按键。
2. 网络传输层：WiFi、MQTT、AP 本地热点。
3. 平台服务层：Node-RED、数据库、HTTP API、告警和控制服务。
4. 应用展示层：Web 端和 uni-app 移动端。

### 3.2 总体运行流程
1. ESP32 定时采集传感器数据。
2. 本地完成基础判断，如满载、异常气体、倾倒等。
3. 通过 MQTT 将数据上报 Node-RED。
4. Node-RED 处理后写入数据库并同步到前端。
5. Web 和移动端通过 HTTP API 查询实时数据和历史数据。
6. 用户发出控制请求后，后端通过 MQTT 下发至 ESP32 执行。
7. 当网络不可用时，ESP32 切换 AP 模式，提供本地查询和开锁接口。

## 4. 模块设计

### 4.1 感知控制模块

职责：完成环境感知、容量检测、姿态监测、身份靠近检测和执行器控制。

输入：
1. DHT11、MQ-135、HC-SR04、HC-SR501、MPU6050 的传感器信号。
2. 来自键盘或后端的控制指令。

输出：
1. 温湿度、气体浓度、容量、人体靠近、姿态状态等采集数据。
2. 风扇、门体、LED、蜂鸣器的执行结果。

异常：
1. 传感器数据超量程时标记无效值。
2. 姿态异常时触发防盗告警。
3. 满载时禁止继续投放并触发锁定逻辑。

依赖：
1. ESP32 GPIO、ADC、I2C 接口。
2. 本地任务调度和消息队列。

对接对象：
1. MQTT 上报模块。
2. OLED 显示模块。
3. 后端控制模块。

### 4.2 网络通信模块
职责：负责 ESP32 与后端之间的 WiFi 和 MQTT 通信，以及离线 AP 模式切换。

输入：
1. 网络配置参数。
2. 采集模块生成的数据。
3. 后端下发的控制消息。

输出：
1. 上报至 MQTT Broker 的遥测数据。
2. 上报至后端的告警消息。
3. 本地 AP 模式下的 HTTP 响应。

异常：
1. WiFi 断开后自动重连。
2. 重连失败后进入 AP 模式。
3. MQTT 连接失败时缓存关键状态并重试。

依赖：
1. ESP32 WiFi 功能。
2. MQTT 客户端库。
3. WebServer 本地服务能力。

对接对象：
1. Node-RED 后端。
2. 本地移动端用户。

### 4.3 后端处理模块
职责：完成数据接收、解析、存储、查询、控制转发和告警管理。

输入：
1. MQTT 上报的数据和告警消息。
2. Web 与移动端发起的 HTTP 请求。

输出：
1. 数据库中的实时数据与历史数据。
2. HTTP API 响应结果。
3. 转发至设备的控制指令。
4. 告警记录和在线状态。

异常：
1. 数据格式错误时返回失败状态并记录日志。
2. 设备离线时不下发控制，直接提示失败。
3. 数据库写入失败时保留最近缓存数据。

依赖：
1. Node-RED。
2. 数据库。
3. MQTT Broker。

对接对象：
1. ESP32 设备端。
2. Web 前端。
3. 移动端。

### 4.4 Web 展示模块
职责：提供 PC 浏览器端的数据监控和控制界面。

输入：
1. 后端 HTTP API 返回的数据。
2. 实时告警和设备状态。

输出：
1. 图表、卡片、状态灯、控制按钮。
2. 用户控制请求。

异常：
1. 请求失败时显示错误提示。
2. 设备离线时显示不可控状态。

依赖：
1. Vue 或原生前端框架。
2. Chart.js 图表组件。

对接对象：
1. Node-RED HTTP API。

### 4.5 移动端模块
职责：提供手机端实时监控、历史查看和远程控制。

输入：
1. 后端 HTTP API 数据。
2. 本地 AP 模式下的 ESP32 接口数据。

输出：
1. 卡片式状态信息。
2. 历史图表。
3. 控制请求和本地开锁请求。

异常：
1. 网络中断时自动切换到本地模式提示。
2. 接口失败时保留上一次有效数据。

依赖：
1. uni-app。
2. 前端 HTTP 请求封装。

对接对象：
1. Node-RED 后端。
2. ESP32 本地 AP 接口。

### 4.6 告警联动模块
职责：对满载、倾倒、异味异常、密码错误等事件进行统一处理。

输入：
1. 传感器异常数据。
2. 密码错误事件。

输出：
1. 蜂鸣器和 LED 告警。
2. 后端告警上报。
3. 前端弹窗提示。

异常：
1. 同类告警短时间内去重。
2. 告警恢复后自动解除提示。

## 5. 接口定义

### 5.1 ESP32 内部接口

#### 5.1.1 传感器数据结构 `SensorData`
- `float capacity`：箱体满载率，单位 `%`
- `float temperature`：温度，单位 `℃`
- `float humidity`：湿度，单位 `%`
- `uint16_t airQuality`：空气质量值
- `bool humanDetect`：人体接近标志
- `bool boxTiltAlert`：倾倒标志
- `bool fanStatus`：风扇状态
- `bool frontDoorState`：投放门状态
- `bool backDoorState`：维护门状态
- `char netMode[4]`：`STA` 或 `AP`

#### 5.1.2 执行指令结构 `ExecCmd`
- `OPEN_FRONT`
- `CLOSE_FRONT`
- `OPEN_BACK`
- `CLOSE_BACK`
- `FAN_ON`
- `FAN_OFF`
- `ALARM_START`
- `ALARM_STOP`

字段：
- `uint8_t cmdType`
- `uint32_t cmdTime`

#### 5.1.3 告警事件结构 `AlertEvent`
- `ALERT_BOX_FULL`
- `ALERT_BOX_TILT`
- `ALERT_PWD_ERROR`
- `ALERT_AIR_BAD`

字段：
- `uint8_t alertType`
- `char alertMsg[64]`

#### 5.1.4 内部队列约定
1. 传感器数据队列：采集任务写入，显示任务和 MQTT 任务读取。
2. 执行指令队列：MQTT 任务和键盘任务写入，执行器任务读取。
3. 告警事件队列：采集任务写入，告警任务读取。
4. 键盘输入队列：按键扫描任务写入，密码校验任务读取。
5. I2C 互斥信号量：OLED 和 MPU6050 共用。
6. WiFi 互斥锁：STA/AP 模式切换共用。

### 5.5 前端调用约定
1. Web 和移动端统一通过 HTTP API 获取实时数据和历史数据。
2. 页面加载后先请求实时状态，再异步加载历史曲线。
3. 实时页面采用定时轮询刷新，轮询周期建议 5s。
4. 控制接口采用提交后立即返回的方式，不等待设备执行完成。
5. 接口失败时返回统一错误结构，并在前端提示用户。

### 5.8 状态机设计
#### 5.8.1 设备工作状态
1. `INIT`：上电初始化，完成传感器、网络、显示、队列初始化。
2. `STANDBY`：待机监测，周期采集并等待投放或控制。
3. `PUT_IN`：人体检测到靠近后进入投放联动状态，开启投放门。
4. `FULL_LOCK`：满载锁定状态，禁止继续投放并提示清运。
5. `ALARM`：异常告警状态，触发蜂鸣器、LED 和后端告警上报。
6. `MAINTENANCE`：管理员维护状态，允许本地密码开锁和检修。
7. `AP_MODE`：网络异常应急状态，开启本地热点和本地接口。

#### 5.8.2 状态切换规则
1. 上电后从 `INIT` 进入 `STANDBY`。
2. 检测到人体靠近且未满载时，进入 `PUT_IN`。
3. 检测到满载后，进入 `FULL_LOCK`。
4. 检测到倾倒、异味异常或密码错误过多时，进入 `ALARM`。
5. 管理员通过密码验证后，进入 `MAINTENANCE`。
6. WiFi 长时间不可用时，进入 `AP_MODE`。
7. 异常解除且条件恢复后，返回 `STANDBY`。

### 5.9 安全与鉴权
1. 后端控制接口必须携带管理员令牌。
2. 本地开锁密码不明文保存，ESP32 端应保存哈希或等价校验结果。
3. 控制指令必须记录来源，区分 Web、App、Local。
4. 告警和控制日志应保留，便于追溯。
5. 设备 ID、密码、Token 等敏感信息不得硬编码到前端页面。

### 5.10 ESP32 的 NVS 存储
ESP32 侧建议使用 NVS 保存以下内容：
1. WiFi SSID。
2. WiFi 密码。
3. 设备 ID。
4. AP 模式热点名称和密码。
5. 管理员密码哈希或本地校验值。
6. 上次工作模式 `STA/AP`。
7. 传感器阈值配置，如满载阈值、告警阈值。
8. 校准参数，如超声波空箱距离基准值。
9. 最后一次成功联网时间。
10. 固件版本号或设备配置版本号。

### 5.11 联调约定
1. 时间统一使用 Unix 时间戳或标准北京时间字符串，后端转换后入库。
2. 设备编号由后端和嵌入式共同确认，格式保持唯一且固定。
3. MQTT 主题名、HTTP 路径、JSON 字段名在开发期间不得随意修改。
4. 轮询接口返回必须稳定，避免前端页面因字段缺失报错。
5. 告警和控制必须有日志，便于联调和排错。

## 6. 开发对接说明
1. 嵌入式工程师可根据感知控制模块和内部接口直接拆分采集、执行、告警、网络四个任务。
2. 后端工程师可根据 MQTT Topic 和 HTTP API 直接搭建 Node-RED 流程与数据库结构。
3. Web 和移动端工程师可根据接口定义直接实现页面、图表、控制按钮和告警提示。
4. 联调阶段以 MQTT 消息、HTTP 返回值和设备状态变化作为统一验收依据。
