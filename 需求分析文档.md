# 智能衣物回收箱物联网系统需求分析文档

## 1. 项目背景
本项目面向社区旧衣物回收、校园回收点和物业服务点等应用场景，设计一套基于 ESP32 的智能衣物回收箱物联网系统。系统通过传感器采集箱体环境和状态信息，通过 MQTT 和 HTTP 与后端及前端联动，实现本地控制、远程控制、数据存储、历史查询与可视化展示，并支持断网后的本地应急管理。

## 2. 系统应用场景
### 2.1 社区旧衣物回收箱
在社区回收点，系统用于监测箱体满载情况、箱内空气质量、温湿度和倾倒风险，帮助运维人员及时清运和维护。

### 2.2 校园或公共场所回收点
在校园或公共场所，系统用于支持无人值守回收、远程查看设备状态、接收异常告警，提升管理效率。

### 2.3 断网应急场景
当后端服务不可达或外网中断时，ESP32 自动进入 AP 热点模式，用户可通过手机直连设备查看状态并进行本地开锁操作。

## 3. 目标用户
1. 运维管理员：负责查看设备状态、处理告警、执行远程控制和现场维护。
2. 实训开发人员：负责嵌入式采集、后端接口、Web 页面、手机端页面和联调测试。

## 4. 系统实现的具体功能

### 4.1 本地控制功能
1. 支持管理员通过 4×4 矩阵键盘输入密码进行本地维护门解锁。
2. 支持本地声光提示，包括蜂鸣器告警和 LED 状态指示。
3. 支持本地 OLED 显示当前容量、温湿度、空气质量、网络状态和告警状态。
4. 支持断网时通过 ESP32 热点连接设备并访问本地接口。

### 4.2 远程控制功能
1. 支持 Web 端和手机端远程查看设备状态。
2. 支持远程下发开门、关门、风扇启停、告警取消等控制指令。
3. 支持远程查看设备在线/离线状态。
4. 支持远程接收异常告警信息。

### 4.3 数据采集功能
1. 采集箱体内部温湿度。
2. 采集箱内空气质量或异味浓度。
3. 采集箱体剩余容量并判断是否满载。
4. 采集人体靠近状态，用于唤醒投放流程。
5. 采集箱体姿态，用于防倾倒、防盗监测。

### 4.4 数据存储功能
1. 保存实时状态数据，供首页展示和设备状态刷新使用。
2. 保存历史时序数据，供趋势图和统计分析使用。
3. 保存告警数据，供追溯和运维处理使用。
4. 保存控制记录，供审计和联调排错使用。

### 4.5 查询与可视化功能
1. Web 端展示实时仪表盘、历史曲线、告警列表和控制面板。
2. 手机端展示实时卡片、历史趋势图、告警提示和控制按钮。
3. 支持按设备编号和时间范围查询历史数据。
4. 支持对温湿度、空气质量、容量等关键指标进行趋势可视化。

## 5. 硬件需求

### 5.1 主控与通信
1. ESP32 开发板，作为系统主控。
2. WiFi 通信能力，支持 STA 联网和 AP 热点模式。

### 5.2 传感器需求
系统至少使用以下 5 类传感器：
1. DHT11 温湿度传感器：采集箱体内部温湿度。
2. MQ-135 空气质量传感器：采集箱内异味或气体浓度。
3. HC-SR04 超声波传感器：检测箱内高度并计算满载率。
4. HC-SR501 人体红外传感器：检测投放人员靠近，用于唤醒投放流程。
5. MPU6050 姿态传感器：检测箱体倾斜或倾倒，触发防盗告警。

### 5.3 执行器需求
系统至少包含以下执行器：
1. OLED 显示屏：显示实时状态、告警和网络信息。
2. 舵机或直流电机：控制投放门或维护门开关。
3. 继电器模块：控制风扇启停。
4. 蜂鸣器：用于声光告警。
5. LED 指示灯：用于显示正常、预警和故障状态。

## 6. 软件功能需求

### 6.1 嵌入式软件需求
1. 定时采集多个传感器数据。
2. 根据人体红外传感器触发投放唤醒流程。
3. 根据超声波数据判断箱体是否满载。
4. 根据 MPU6050 判断是否存在倾斜或倾倒风险。
5. 将采集数据封装后通过 MQTT 上报后端。
6. 接收后端下发的控制指令并驱动执行器。
7. 在网络异常时自动切换 AP 模式提供本地服务。

### 6.2 Web 端软件需求
1. 实时显示设备状态、容量、温湿度、空气质量和在线状态。
2. 展示历史数据曲线和告警记录。
3. 提供远程控制入口，如开门、关门和风扇控制。
4. 支持设备状态轮询刷新。

### 6.3 手机端软件需求
1. 显示设备实时状态卡片。
2. 显示历史趋势图。
3. 支持远程控制和告警提醒。
4. 支持离线模式下连接 ESP32 热点查看状态和本地开锁。

## 7. 传感器唤醒与交互逻辑
1. HC-SR501 检测到人员靠近后，系统进入投放唤醒状态。
2. 投放唤醒后，ESP32 控制舵机打开投放门，同时 OLED 和 LED 显示当前可投放状态。
3. 投放过程中，HC-SR04 持续参与容量检测，判断是否满载。
4. DHT11 和 MQ-135 周期采集环境数据并同步给后端，用于监测是否需要通风。
5. MPU6050 持续监测箱体姿态，若检测到倾斜或倾倒，立即触发声光告警并上报后端。
6. 传感器采集结果通过 MQTT 发送给后端，后端再同步到 Web 端和手机端。
7. Web 端和手机端可根据实时状态发起控制请求，后端转发后由 ESP32 执行。

## 8. 非功能需求
1. 实时性：采集周期约 2s，控制响应应尽量在 1s 内完成。
2. 稳定性：系统应支持长时间连续运行，避免阻塞和死锁。
3. 容错性：单个传感器异常不影响系统整体运行。
4. 兼容性：支持 PC 浏览器和手机端访问。
5. 可维护性：模块划分清晰，便于嵌入式、后端和前端分别开发。
6. 可扩展性：后续可增加更多传感器、更多控制策略和更多告警规则。

## 9. 需求总结
本系统需求可概括为“感知采集、状态判断、本地控制、远程控制、数据存储、查询可视化、断网应急”七个核心目标。系统通过 ESP32、传感器、执行器、Node-RED、Web 端和手机端的协作，形成完整的智能衣物回收箱物联网闭环。
