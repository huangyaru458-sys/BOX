# 需求分析文档

## 1. 场景说明
本系统用于社区旧衣物回收、校园回收点和物业场景，实现箱体状态监测、远程控制、告警联动和离线应急管理。

## 2. 角色需求
1. 管理员：查看状态、处理告警、下发控制、查看历史。
2. 维护人员：本地开锁、查看设备状态、处理异常。
3. 普通用户：查看设备状态和告警信息。

## 3. 功能需求
### 3.1 监测
1. 采集温湿度、空气质量、容量、人体接近、姿态倾斜等数据。
2. 超声波传感器使用 `Trig GPIO2`、`Echo GPIO35`。
3. 数据需与 ESP32 串口、MQTT、OLED、Node-RED 调试窗口保持一致。

### 3.2 本地控制
1. 4x4 矩阵键盘支持本地维护开锁。
2. 前门为 `GPIO33` 灯光模拟，不再使用舵机。
3. 后门为 `GPIO12` 灯光模拟，不再使用舵机。
4. 风扇由 `GPIO27 + GPIO15` 驱动。
5. 蜂鸣器由 `GPIO5` 控制。
6. 键盘支持快捷开门、关风扇、取消告警。

### 3.3 远程控制
1. Web 端可控制前门、后门、风扇和告警取消。
2. 远程控制通过 Node-RED 图形化流程转发到 MQTT。
3. 设备离线时必须拒绝控制并返回明确错误。
4. 控制结果需有状态回传，前端状态不能只显示“执行中”。

### 3.4 数据存储
1. 实时状态需要持久化。
2. 历史数据需要保存，支持曲线回看。
3. 告警记录需要保存，支持查询与已读处理。
4. 控制记录需要保存，支持审计。

### 3.5 可视化
1. Web 端展示实时状态、历史曲线、告警列表和设备列表。
2. Dashboard 端展示常用控制、状态摘要和调试信息。

### 3.6 网络与稳定性
1. WiFi 断开后自动重连。
2. MQTT 断开后自动重连。
3. 后端不可用时进入 AP 离线模式。
4. 设备离线时不允许下发硬件控制。

## 4. 非功能需求
1. 控制响应要快，页面点击后应尽快反馈。
2. 系统需支持长期运行。
3. 单个传感器异常不应拖垮整体。
4. 后端重启后数据仍可查询。

## 5. 接口需求
### 5.1 MQTT
- 上报：`device/clothes_box/telemetry`
- 告警：`device/clothes_box/alert`
- 控制：`device/clothes_box/control`
- 应答：`device/clothes_box/control_ack`

### 5.2 HTTP
- `GET /api/v1/box/status`
- `GET /api/v1/box/history`
- `GET /api/v1/box/devices`
- `GET /api/v1/box/alerts`
- `GET /api/v1/box/commands`
- `GET /api/v1/box/health`
- `POST /api/v1/box/control`
- `POST /api/v1/box/alerts/read`

兼容路由：
- `GET /api/data/latest`
- `GET /api/data/history`
- `GET /api/devices`
- `POST /api/control`

## 6. 数据库需求
后端必须有 SQLite 图形化节点链路，至少包含：
1. 建表节点
2. 遥测写库节点
3. 告警写库节点
4. 控制记录写库节点
5. 历史查询节点
6. 设备列表节点

核心表：
- `devices`
- `device_latest_status`
- `device_history`
- `device_alerts`
- `device_commands`
- `telemetry`
- `alerts`
- `commands`
- `admin_users`

## 7. 验收项
1. 前门可开关。
2. 后门可开关。
3. 风扇可开关。
4. 蜂鸣器可控制。
5. 告警可取消并抑制 5 分钟。
6. 离线时禁止操作硬件。
7. 数据库可落库。
8. 历史曲线可回看。
9. MQTT 可连接并自动重连。
10. 传感器数据显示与后端、串口、OLED 一致。
